  1. 高保真炼金合成概率模拟（预测 + 批量仿真）

- **当前阶段**：基础框架已完工，进入功能完善与自动化阶段

## 一、OCR → 模型自学习闭环

### 1.1 Codex Skill：自动图文识别录入

**目标**：在 Codex 中做一个 Skill，实现全自动化管道：

```
拉取 pending 图片 URL → Agent 图文识别 → 去重过滤 → 可信度比对 → 入库
```

**具体步骤**：
1. 读取 `uploads.json`，筛选 `status: "pending"` 的记录
2. 将图片 URL 发送给 Vision Agent 做 OCR 识别
3. **去重**：与已有 `recognized` 记录的配方比对，剔除重复配方
4. **可信度比对**：
   - 检测无效图片（非合成截图、模糊不清等）
   - 将识别结果与 `alchemy_core.js` 已有模型交叉验证
   - 标记低可信度结果，人工复核
5. 通过 `import_results.py` 逻辑回填数据库

**后期**：Codex 定时任务自动执行。

### 1.2 预测校准与验证

**现状**：模拟器已支持 `白玉 + 粉水晶 → xx% 白玉` 概率预测，但基于理论模型。

**待做**：
- 用收集到的真实配方数据与理论预测做比对
- 标记偏离项（"已包含在模型中" / "待学习/新配方"）
- 修正概率权重（目前固定分布：+4:10%, +3:25%, +2:30%, +1:15%, +0:15%, 降级:5%）

### 1.3 配方数据用途

- ✅ **发现缺失配方**：模型未覆盖的合成路径
- ✅ **校准概率权重**：用真实数据修正理论概率分布

## 二、模拟器功能进化

### 2.1 多材配方支持（2~5 槽）

**现状问题**：
- 模拟器已支持 2~5 槽 ✅
- 但 `upload.html` 标注表单只有 Slot1 + Slot2 ❌
- `import_results.py` 的 `RecipeEntry` 也只建模了 2 个材料位 ❌

**待升级**：
- `RecipeEntry` 结构体扩展为支持 Slot3~Slot5（可选字段，向后兼容）
- `upload.html` 表单支持动态添加副材位
- `import_results.py` 自动补全材质时覆盖全部槽位
- `export_pending.py` 的 Prompt 模板同步更新

### 2.2 性能优化

**痛点**：批量模拟 1000 次在前端同步执行，大数据库下可能卡 UI。

**方案**（记录为优化路径）：
- 将批量仿真逻辑移入 Web Worker
- Worker 中载入 `alchemy_core.js`，独立线程执行
- 主线程通过 `postMessage` 接收进度和结果

### 2.3 数据持久化

**现状**：`localStorage` 保存历史（最多 100 条），换设备丢失。

**策略**：
- 短期：保持 localStorage，够用
- 中期（待评估）：可选导出/导入 JSON
- 远期（待评估）：账户系统 + 服务端存储

### 3.2 Go 后端分层重构

**现状**：单文件 `main.go`，混合了配置加载、COS 上传、路由、数据持久化。

**目标分层**：
```
uploader_service/
├── main.go              # 入口，启动服务
├── config/
│   └── config.go        # 配置加载
├── handler/
│   ├── upload.go        # POST /api/upload
│   └── records.go       # GET /api/upload/records
├── store/
│   └── json_store.go    # uploads.json 读写 + 互斥锁
├── cos/
│   └── client.go        # 腾讯云 COS 操作封装
└── middleware/
    └── cors.go          # CORS 中间件
```
